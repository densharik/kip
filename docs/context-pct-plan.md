# План: моментальный % контекста сессии (redo с нуля, после eng review)

## Факты (проверены живыми данными 2026-07-23)
- Готовый % нигде не хранится. Claude Code считает его в рантайме и раз в ~1с отдаёт в statusline-хук (`context_window.used_percentage` в stdin-JSON вместе с `session_id`).
- Сырьё на диске универсально для всех пользователей: `~/.claude/projects/<enc-cwd>/<sid>.jsonl` дописывается usage-токенами после каждого ответа API; `~/.claude/sessions/<pid>.json` - метаданные живого процесса (sessionId, cwd, name).
- % закрытой сессии заморожен => известен ДО запуска claude, если известен id.
- Пикер (`--resume` без id): id физически не существует до выбора; старт CLI ~1.3с - пол.
- statusline-команда запускается самим claude => ppid хука = pid claude.

## Цели
1. Работает у ЛЮБОГО пользователя rwarp - без claude-rpg и другой личной обвязки. Никаких захардкоженных путей, всё от home_dir; деградация без хука - плавная.
2. Resume с известным id (кнопка, введённая команда, алиас): % на экране <=100мс от Enter, до старта claude.
3. Живая сессия: обновление после каждого ответа API (<=0.5с), а не раз в 5-8с.
4. Не зависеть от cwd терминала (резюм чужого проекта из любой папки).

## Дизайн

### 0. Поток данных

```
                 [claude process]
                   |          \
        пишет jsonl|           \ раз в ~1с зовёт statusline
                   v            v
  ~/.claude/projects/**/  [rwarp-ctx-hook] --wrap--> [прежний statusline юзера]
      <sid>.jsonl              |
  ~/.claude/sessions/          v  атомарно tmp+rename
      <pid>.json          ~/.rwarp/ctx/by-sid/<sid>.json   {session_id, ctx, ts}
                          ~/.rwarp/ctx/by-pid/<pid>.json
                   \           |
        фоновые потоки: lookup / оценка / поллер 500мс (stat)
                    \          |
                     v         v
              mpsc-канал (sid, pct, exact, source_ts)
                         |
                         v  дрен в начале кадра
              CtxIndex: HashMap<sid, {pct, exact, source_ts}>  (владеет UI)
                         |
                         v  синхронный lookup, 0мс
                  бейдж в строке сессии
```

### 1. rwarp-ctx-hook - универсальный точный источник
Крошечный POSIX-скрипт (#!/bin/sh, без bash-измов - /bin/sh на macos бывает и bash, и zsh).
Живёт НЕ в бандле: при включении галочки копируется из Contents/Resources в `~/.rwarp/bin/rwarp-ctx-hook.sh`,
в settings.json прописывается через `"$HOME"` - переживает перенос/обновление/дубли .app и dotfiles-синк settings.json на другую машину.
Механика (qwen-ревью: stdin - поток, читается один раз, "exec с тем же stdin" в лоб невозможен):
1. `mkdir -p` каталогов при каждом вызове; 2. stdin буферизуется во tmp-файл целиком;
3. извлечение полей БЕЗ jq/python (их нет в чистой macos): tr -d '\n' + sed по двум машинным полям -
   session_id (UUID-чарсет) и used_percentage (число); общий JSON-парсер не нужен;
4. атомарно (tmp+rename) пишет `{session_id, ctx, ts}` в `~/.rwarp/ctx/by-sid/<sid>.json` и `~/.rwarp/ctx/by-pid/<ppid>.json` (ppid = pid claude; claude зовёт команду как одиночную -> shell делает exec-оптимизацию, проверено живыми данными);
5. при обёртке - `exec prev_cmd < tmpfile` (настоящий exec, stdin из буфера); её stdout уходит claude без изменений;
6. ЛЮБАЯ ошибка записи снапшота (диск, права) -> молча пропускаем и всё равно запускаем prev_cmd: statusline юзера первичен, наши данные вторичны.
Установка - только по явному действию: галочка "Точный % контекста (statusline-хук)" в настройках rwarp:
- statusLine в `~/.claude/settings.json` отсутствует -> прописываем хук;
- statusLine уже есть (чужой скрипт) -> оборачиваем: прежняя команда сохраняется в конфиге rwarp и передаётся хуку;
- settings.json правится через serde_json::Value (трогаем ТОЛЬКО ключ statusLine, остальное байт-в-байт) + tmp+rename;
- откат при снятии галочки: восстанавливаем прежнюю команду ТОЛЬКО если текущий statusLine всё ещё наш хук; юзер сменил вручную -> не трогаем вообще;
- ctx=0/отсутствует -> снапшот не пишем (нет данных).
Уборка: фоновый sweep при старте rwarp (паттерн sweep_paste_temp): by-pid старше 24ч, by-sid старше 30 дней - удаляются.
Претензия Codex "PID reuse / несколько claude": by-pid записи отсекаются по ts (>10с - мусор); фоновые процессы различаются по своим PTY (fg_pid каждой вкладки свой); два claude на ОДНОМ sid - см. приоритет by-pid в §4.

### 2. CtxIndex - единственный источник правды для UI (src/ctx_index.rs)
`HashMap<sid, CtxEntry { pct: f32, exact: bool, source_ts: SystemTime }>` - владеет App (UI-поток).
Обновления ТОЛЬКО через mpsc-канал (4A) из фоновых потоков - как git/stats/имена; в UI ни одного лока и ни одного файлового чтения.
Merge (1A + уточнение Codex): побеждает запись с более свежим SOURCE_TS - временем ИСТОЧНИКА (ts из снапшота хука; mtime jsonl для оценки), НИКОГДА не временем чтения - поздний результат фонового скана не может перетереть актуальное. При равенстве (+-2с) - exact.
Lookup(sid) при промахе (фоновый поток, 6A - никакого стартового скана). НЕ короткое замыкание, а ОБА источника в merge (qwen-ревью: снапшот прошлого запуска может стухнуть - compact/продолжение на другой машине, свежий jsonl обязан его перебить):
1. читает `~/.rwarp/ctx/by-sid/<sid>.json` напрямую (путь выводится из sid; лимит 64КБ) -> в канал с source_ts = ts снапшота;
2. И оценка по jsonl: путь из карты sid->jsonl; tail 64КБ, при отсутствии валидной usage-строки - прогрессивно до 1МБ (реальная строка jsonl бывает >118КБ - замерено; одна гигантская строка не должна давать ложное "нет данных"); ОБРАТНЫЙ скан до первой валидной usage-строки (оборванная при дозаписи пропускается - Codex-пункт закрыт); source_ts оценки = поле `timestamp` САМОЙ usage-записи (есть в каждой строке, проверено), НЕ mtime файла - иначе дозапись user-message затирала бы exact устаревшей оценкой; окно: эвристика (fable/mythos/[1m] -> 1M, иначе 200k) + самокалибровка - если (input+cache) уже больше предполагаемого окна, окно повышается до 1M;
3. ничего не нашли - записи нет, бейджа нет (не 0%).
sid из снапшотов/имён файлов валидируется valid_sid (машинный источник); строгий valid_uuid - только для пользовательского ввода (parse_resume_hint).
Все пути `~/.claude/**` строятся от `$CLAUDE_CONFIG_DIR`, если переменная задана, иначе от home_dir (и в rust-коде, и в хуке).

Карта sid->jsonl (3A, жизненный цикл - Codex-пункт закрыт): один фоновый обход ИМЁН файлов `~/.claude/projects/*/*.jsonl` при старте (без чтения содержимого); промах -> пересканирование не чаще 1/30с; новые привязки добавляют путь в карту сразу (project_dir(cwd) - первый кандидат); карту читает и поллер.

### 3. Моменты привязки id (код уже существует)
- Enter в rwarp: parse_resume_hint (`--resume <uuid>`/`-r`/`-r<uuid>`/`--resume=`/`--continue`; строгий UUID - `--resume` принимает и поисковый термин) -> lookup сразу.
- Кнопка "Продолжить Claude", старт приложения (восстановленные сессии) -> lookup.
- Алиасы/скрипты: argv claude (`ps -ww -o args=`, -ww обязателен) при детекции по первому выводу PTY (~0.2-0.4с).
- Пикер и `--continue`: скоринг `~/.claude/sessions/<pid>.json` (файлы самого Claude Code - РАБОТАЕТ БЕЗ нашего хука; Codex тут ошибся) + by-pid снапшот хука; burst-поллинг 300мс первые 3с после появления claude.

### 4. Прямой эфир (сессия работает)
mtime+size-поллер 500мс - ОТДЕЛЬНЫЙ таймер (поле last_ctx_stat) внутри housekeeping, НЕ под общим `if >= TICK` (TICK=2с, main.rs:24 - иначе поллер молча станет двухсекундным); только для живых claude-сессий, только stat (чтение - лишь при изменении), всё в фоне:
- `~/.rwarp/ctx/by-pid/<fg_pid>.json` изменился -> exact в канал; для вкладки с известным fg_pid это ПРИОРИТЕТНЫЙ источник (два claude на одном sid: by-sid - last-writer-wins и общий на всех, by-pid - персональный для процесса вкладки); sid ВНУТРИ отличается (2A) -> перепривязка вкладки + свежий % (<=1.5с при /resume внутри claude);
- `~/.rwarp/ctx/by-sid/<sid>.json` изменился -> exact в канал (для вкладок без fg_pid и закрытых сессий);
- `<sid>.jsonl` изменился -> оценка немедленно (скачок % в момент ответа API, раньше хука; перекроется exact по source_ts).
Ошибка чтения/файл исчез -> запись в индексе не трогаем, повтор следующим тиком. kqueue/FSEvents не берём (выигрыш <=0.5с, не пробивает пол пикера).

### 5. Семантика бейджа (Codex-пункт закрыт)
% из индекса по s.claude_session_id. Нет записи - нет бейджа (никаких 0%). Закрытая сессия - замороженный % и есть истина. Цвета: серый / оранжевый >=50 / красный с пульсом >=70 (пульс 10fps, только при видимом бейдже).

- hook-скрипт: смок-тест sh -c с фикстурным stdin (валидный/пустой/без ctx; проверка что prev_cmd получает полный stdin из буфера).
